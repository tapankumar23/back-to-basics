Financial Systems

Reservations, payments, wallets, and exchange matching engines

9.1 — Hotel Reservation Race conditions, optimistic locking, idempotency
Optimistic locking scales well — no locks held between read and write. DB constraints (CHECK available >= 0) are a safety net. Idempotency keys prevent double-booking from network retries by detecting duplicate requests before processing.
9.2 — Payment System Pay-in flow, double-entry bookkeeping, reconciliation
Double-entry bookkeeping ensures every dollar is accounted for — every debit has a matching credit. Reconciliation catches discrepancies between internal ledger and external payment processor records (PSP). Idempotency keys on webhooks prevent duplicate payment processing.
9.3 — Digital Wallet Event sourcing, 2-phase commit, saga pattern
Event sourcing: the balance is not stored directly — it is derived by replaying all events, giving a complete audit trail. 2PC provides strong consistency but blocks on coordinator failure. Saga is resilient but requires careful compensation logic to undo partial transactions.
9.4 — Stock Exchange Order book, matching engine, LMAX Disruptor ring buffer
The order book is sorted by price-time priority. The matching engine is the heart of the exchange — it must process millions of orders per second. The LMAX Disruptor's lock-free ring buffer eliminates CPU cache invalidation from lock contention, achieving microsecond latency.
