❌ Inheritance
class GuideDog extends Dog {
    // Must inherit ALL of Dog
    // RoboticGuideDog cannot extend
    // Dog — hierarchy breaks
}
✅ Composition
class GuideDog {
  private final GuideAbility guide
    = new GuideAbility();
  void guide(Person p) {
    guide.execute(p);
  }
}
class RoboticGuideDog {
  private final GuideAbility guide
    = new GuideAbility();
  // same behaviour, no hierarchy
}
Inheritance means a subclass is a specialisation of its parent — it inherits everything. This causes problems when you need to share behaviour across unrelated classes. A RoboticGuideDog and a GuideDog both guide, but one is not a mammal. Composition solves this: each class holds behaviour objects and delegates to them.
Use inheritance for true IS-A relationships with stable shared contracts (a Square IS-A Shape). Use composition for behaviour sharing (a Dog HAS-A SwimAbility). Inheritance is a strong compile-time coupling. Prefer it sparingly; favour interfaces and composition for flexibility.

❌ LoD Violation
String zip = order
  .getCustomer()
  .getAddress()
  .getCity()
  .getPostalCode();
// 4 objects exposed internally
✅ LoD Compliant
String zip =
  order.getShippingPostalCode();
// ONE arrow, one dependency.
// Order encapsulates the path.
Count the dots in a method chain. More than one dot is usually a Law of Demeter violation — you are depending on the internal structure of objects you should not know about. Each extra dot is a coupling to a different class's internals.
DTOs, records, and value objects are exempt — accessing point.x or address.city is fine because DTOs are transparent containers, not objects with behaviour. Fluent interfaces (builders, streams) are intentional LoD exceptions — stream.filter(...).map(...).collect(...) is fine because the chain is a designed API and each method returns the same type.
4.3 — Tell Don't Ask Delegate decisions to the object that owns the data
Coupling comparison:
❌ Asking externally
if (customer.getMembershipLevel()
    .equals("GOLD") &&
    customer.getTotalSpend() > 1000 &&
    !customer.isBlacklisted()) {
  order.applyDiscount(0.15);
}
// OrderService knows internals
✅ Telling the object
customer
  .applyLoyaltyDiscountTo(order);
// Customer decides for itself
// OrderService has zero knowledge
// of membership/spend/blacklist
Instead of asking an object for its data and making decisions about it yourself, tell the object what you want done and let it use its own data to decide. The object that owns the data is the right place for the logic that acts on it.
TDA and CQS can appear to conflict. Reconcile them: TDA is about behaviour delegation (telling an object to act), CQS is about return-type discipline (commands vs queries). They operate at different levels and work together. TDA leads to richer domain models since logic naturally migrates to where the data lives.

QUERY — easy to test
Pure function. Same input, same output. No setup, no teardown, no side-effect mocking.
COMMAND — test side effects
Verify state changed. No return value confusion. Each test checks one thing.
❌ Mixed
List<Order> processAndReturn(
    List<Order> pending) {
  pending.forEach(o -> {
    o.setStatus(PROCESSED);
    db.save(o); // mutation!
  });
  return pending; // also returns
}
// Reader triggers side effects
✅ Separated
// COMMAND: mutates, void return
void processOrders(
    List<Order> pending) {
  pending.forEach(o -> {
    o.setStatus(PROCESSED);
    db.save(o);
  });
}
// QUERY: returns, no mutation
List<Order> getOrders(
    OrderStatus s) {
  return db.findByStatus(s);
}
Ask yourself: does this method change anything? If yes, return void. Does it return data? If yes, change nothing. Mixing both makes code hard to reason about — calling a getter can have hidden consequences.
Some operations are legitimately both — pop() from a queue is atomic and intentionally combined. These are explicit documented exceptions, not rule breaks. CQRS (Command Query Responsibility Segregation) is the architectural scale-up: separate read and write models entirely, often with different databases.

✅ Predictable
class Order {
  String toString() {
    return "Order#" + id;
    // pure, no side effects
  }
  void save() {
    db.insert(this);
    // only saves
  }
  void saveAndNotify() {
    save();
    emailService
      .sendConfirmation(customer);
    // explicit name!
  }
}
Ask: "Would a teammate who has not read this code be surprised by what it does?" If yes, either rename the method so the full behaviour is obvious, or split it into separately-named methods. Method names are contracts.
POLA is context-sensitive. The += operator for string concatenation is surprising in most languages, perfectly normal in Python. The >> operator for stream output is idiomatic C++ but astonishing in Java. Know your language and team conventions — violations are relative to context and shared expectations.
