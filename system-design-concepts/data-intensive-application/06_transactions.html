Transactions

ACID guarantees, isolation levels, concurrency anomalies, and the path to serializability — the building blocks of reliable data systems.

ACID Properties

Transactions promise four guarantees. Together they let you treat a group of reads and writes as a single logical unit.

Atomicity

All-or-nothing: every operation in the transaction succeeds together, or all are rolled back on failure.

Single-Object vs Multi-Object Operations

Some guarantees apply within a single object. Multi-object transactions are needed when multiple records must stay in sync.

Single-Object Atomicity

Even simple operations can have race conditions. Atomic DB operations sidestep the problem.

Read-Modify-Write (Counter)
1. READ x=50
→
2. x=x+1=51
→
3. WRITE x=51
Compare-And-Swap (CAS) — atomic
UPDATE counter
  SET val = val + 1
  WHERE id = 42
Multi-Object Need

When records reference each other, partial updates cause inconsistency. Transactions keep them in sync.

Insert email + update unread counter
✓ INSERT INTO emails(msg)
✗ UPDATE counters SET unread+1 -- FAILS
Email inserted, counter not updated → inconsistency
Idempotency key — safe network retry
INSERT INTO payments
  (id, amount, idem_key)
  VALUES (1, 100, 'key-abc')
ON CONFLICT (idem_key)
  DO NOTHING -- safe retry

Preventing Lost Updates

A lost update happens when two transactions read-modify-write the same object concurrently. The second write clobbers the first.

The Race Condition
Tx A
read=50
compute 51
write 51
Tx B
read=50
compute 51
write 51
Result: 51 — should be 52. Tx A's increment was lost!
Atomic Write

Express the entire read-modify-write as a single atomic DB operation. No separate READ step means no race.

-- DB handles atomically. No separate READ.
UPDATE counter
  SET val = val + 1
  WHERE id = 42
✓ No race condition ✓ Ideal for simple increments
Explicit Lock (SELECT FOR UPDATE)

Acquire a row lock before reading. Other transactions block until you commit or rollback.

BEGIN
SELECT val FROM counter WHERE id=42
  FOR UPDATE  -- acquires exclusive row lock
-- Tx B blocks here until Tx A commits
UPDATE counter SET val=val+1 WHERE id=42
COMMIT
Tx A: 🔒 lock held Tx B: ⌛ waiting…
Automatic Detection (PostgreSQL Repeatable Read)

DB detects the lost update and aborts one conflicting transaction. App retries and gets correct result.

Tx A
read=50
write=51
COMMIT ✓
Tx B
read=50
write=51
ABORT
Tx B retry
read=51
write=52
COMMIT ✓
Final value: 52 ✓ — both increments preserved
Application-Level (Multi-Leader / CRDT)

In multi-leader or leaderless systems the application merges concurrent writes. Simple union often still loses updates for counters — use CRDTs.

Replica A
write: val=51
Replica B
write: val=51
Naive merge (last-write-wins): 51 — still lost an increment!
CRDT G-Counter: 51+51=52 ✓

Write Skew & Phantoms

Write skew: each transaction's write is individually valid, but together they violate a multi-row constraint. Phantoms arise from inserts that change search results mid-transaction.

Doctor On-Call Write Skew
Alice
on call
Bob
on call
Step 1: Tx A reads 2 doctors on call → withdraws Alice
Step 2: Tx B reads 2 doctors on call → withdraws Bob
Step 3: Both commit → 0 doctors on call!
Phantom Reads & Materializing Conflicts

Both transactions check the same predicate, both get "free", both insert — double booking.

Tx A
check 12pm free=yes
book 12pm
Tx B
check 12pm free=yes
book 12pm
Double-booking! Neither saw the other's insert.
Fix: Materialize Conflicts (pre-create slot rows)
-- Pre-populate all possible time slots
INSERT INTO time_slots (time)
  VALUES ('12:00'), ('13:00'), ...

-- Both Txs lock the same slot row
SELECT * FROM time_slots
  WHERE time='12:00' FOR UPDATE
-- Second Tx waits -> serialized

Serializability

The strongest isolation level. The outcome is equivalent to running all transactions one at a time in some serial order.

Actual Serial Execution

One transaction at a time on a single thread. Correct by definition. Redis and VoltDB use this. Requires short transactions and data in RAM.

Two-Phase Locking (2PL)

Growing phase: acquire shared (S) or exclusive (X) locks. Shrinking phase: release only after commit/rollback. Can deadlock — DB must detect and abort a victim.

Lock state
Row Tx A Tx B
row_x X-lock ⌛ wait

Deadlock cycle
Tx A holds row_x, waits for row_y
↕
Tx B holds row_y, waits for row_x
Deadlock detected → Tx B aborted → Tx B retries
Serializable Snapshot Isolation (SSI)

Optimistic: transactions run without blocking. At commit time the DB checks if the read premise was invalidated by a concurrent write. Used in PostgreSQL ≥9.1 and CockroachDB.
