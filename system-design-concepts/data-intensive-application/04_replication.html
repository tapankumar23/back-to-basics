Data Replication

Keeping copies of data on multiple machines for fault tolerance, scalability, and low latency — and all the tricky consistency problems that follow.

Section 1

Single-Leader Replication

One node (the leader) accepts all writes. Followers replicate the leader's write-ahead log. Reads can go to any replica.

Write Path: Leader → WAL → Followers

Replication Log Format

-- Statement-based replication
INSERT INTO orders (id, amount)
VALUES (42, 99.99);
-- Problem: non-deterministic fns (NOW(), RAND())
-- WAL Shipping (physical)
LSN 00000001/0B020A10
page 0x1A2F offset 88 len 56
-- Tightly coupled to storage engine version
-- Row-based (Logical) replication
{"op": "INSERT",
 "table": "orders",
 "new": {"id": 42, "amount": 99.99}}
-- Engine-agnostic; used by Debezium/CDC

Section 3

Replication Lag Anomalies

Async replication means followers can lag. Three classic consistency anomalies emerge.

1. Read-Your-Own-Writes

User posts a comment, refreshes — comment is gone (read from lagging follower).

Route reads to the leader for 60 seconds after a user's write, OR track the replication offset and only read from a follower that has caught up to that LSN.

2. Monotonic Reads

User reads v2 from F1, then gets routed to slower F2 and reads v1 — time goes backwards.

Sticky routing: always send the same user's reads to the same replica. On failure, re-assign only after that replica's successor has caught up.

3. Consistent Prefix Reads

Causally related writes arrive out of order — observer sees the answer before the question.

Route causally related writes to the same partition so they always arrive in order, OR use version vectors to track and enforce causal dependencies.
Section 4

Multi-Leader Replication

Multiple nodes accept writes. Enables multi-datacenter deployments and offline clients — but write conflicts become inevitable.

Dual-Datacenter Write Flow

Write Conflict Detected!
Key user#42.name written simultaneously in DC1 ("Alice") and DC2 ("Alicia").

Conflict Resolution Strategy

// Last-Write-Wins (LWW)
timestamp(DC1) = 1710000001.230
timestamp(DC2) = 1710000001.198
winner = "Alice" // DC1 wins
// ⚠ Clock skew can choose wrong winner!
// Application-level merge
DC1: "Alice", DC2: "Alicia"
merged = ["Alice", "Alicia"]
// App presents both; user picks one
// CRDT (Grow-Only Set example)
DC1.add("Alice"); DC2.add("Alicia");
merged = GSet{"Alice", "Alicia"}
// Automatically convergent — no conflict!

Any node accepts reads and writes. Consistency guaranteed via quorums: when W + R > N there is always at least one overlapping node.

Sloppy Quorum & Hinted Handoff

When N2 and N3 are down, writes are accepted by N1 and N4 with hints. When nodes recover, hints are replayed to bring them up to date.
