Key-Range Partitioning

Assign continuous ranges of keys to each partition — like encyclopedia volumes.

How it works

The keyspace is divided into contiguous ranges. Each range is assigned to a partition (and its replicas). The database keeps a sorted structure of range boundaries — looking up a key requires finding its range in O(log n).

Ordered scans are efficient — e.g. fetch all sensor readings between 2024-01-01 and 2024-01-31 in a single sequential read on the owning partition.
Hot-spot risk: timestamp-prefixed keys send all recent writes to the latest partition. Mitigate by prepending the sensor ID or hashing the prefix.
// Key-range lookup (pseudo) function routeKey(key) { for (let [lo, hi, node] of boundaries) { if (key >= lo && key <= hi) return node; } }
02

Hash Partitioning

Apply a hash function to the key; the resulting hash value determines the partition — distributing load evenly.

Consistent hash ring
Click "Route Key" to watch consistent hashing in action.
Consistent Hashing

Both nodes and keys are hashed onto the same ring. A key is owned by the first node clockwise from its hash position. Adding or removing a node only moves a fraction (1/n) of keys.

Virtual nodes (vnodes) place each physical node at multiple ring positions, improving balance and making load redistribution finer-grained.

Hash partitioning destroys key ordering — range scans require querying all partitions (scatter/gather).
// Consistent hash ring lookup function getNode(key) { const h = hash(key); // 0..2^32 const ring = sortedRing; // sorted node hashes const idx = upperBound(ring, h); return ring[idx % ring.length].node; }

Hot Spots & Skewed Workloads

Even perfect hashing can't prevent celebrity-post hotspots — a single key that gets millions of reads or writes per second.

Write load per partition
post:beyoncé
AFTER MITIGATION (10-shard spread)
All writes for a celebrity post land on a single partition.
The Celebrity Problem

Beyoncé posts → millions of followers write reactions simultaneously. The hash of post:beyoncé maps to one partition, overwhelming it despite even key distribution elsewhere.

Mitigation: append a random suffix to hot keys: post:beyoncé:07. Writes spread across N shards. Reads must query all N shards and merge — a controlled fan-out trade-off.
Only a tiny number of keys are hot. Most applications can detect these (e.g. via a hot-key cache or rate-limiter) and apply the splitting technique selectively.
// Split hot key across N shards function hotWrite(key, value, N) { const shard = Math.floor(Math.random() * N); write(`${key}:${shard}`, value); } function hotRead(key, N) { const parts = [...Array(N)].map((_, i) => read(`${key}:${i}`)); return merge(parts); // fan-out reads }
04

Secondary Indexes: Local vs Global

Partitioning by primary key is straightforward — but secondary indexes complicate the picture significantly.

Local (Document-Partitioned) Index

Each partition maintains its own secondary index covering only the documents in that partition. Writing is simple — you only update the index on the partition that holds the document.

Query cost: a secondary-index query must fan out to all partitions (scatter/gather) and merge results. Expensive for read-heavy workloads.

Used by: Elasticsearch, MongoDB, Cassandra.

Global (Term-Partitioned) Index

The secondary index is itself partitioned (by term). All documents with color=red appear in one index partition, regardless of which data partition holds the document.

Read is efficient — query only the relevant index partition. But writes are expensive: updating a single document may touch multiple index partitions across nodes.

Global index updates are often asynchronous — reads may be slightly stale. Used by: DynamoDB GSI, Riak.

Rebalancing Strategies

When nodes are added or removed, partitions must move to maintain even load — without downtime.

Fixed Partitions

Create many more partitions than nodes at startup (e.g. 1000 partitions, 10 nodes → 100 per node). When a node joins, it steals a few partitions from each existing node. The number of partitions never changes — only their assignment.

Partition count should be chosen to comfortably exceed the maximum expected cluster size. Too many partitions adds bookkeeping overhead.

Partition lifecycle
Partition grows past threshold → auto-split.
Dynamic Splitting

Partitions start large and split when they exceed a size threshold (e.g. HBase's 10 GB default). Small partitions can be merged. The number of partitions adapts to data volume automatically.

Similar to how a B-tree splits its nodes when they overflow. Well-suited to key-range partitioning where split point can be chosen at the median key.
An empty database starts with a single partition. All writes go there until the first split — pre-splitting is recommended.
Partitions per node
Each node owns a fixed number of partitions.
Proportional Partitioning

The number of partitions is proportional to the number of nodes — typically a fixed ratio (e.g. 256 partitions per node in Cassandra). Adding a node increases total partitions; each new node randomly chooses split points.

Partition size stays roughly constant as the cluster grows, since more nodes means more (but smaller) partitions covering the same total data.

This approach sits between fixed (count never changes) and dynamic (count grows with data). Used by: Cassandra, Ketama.

06

Request Routing

When a client wants to read or write, how does it know which node to contact? Three main approaches.

Client-Aware Routing

The client maintains a local copy of the partition-to-node mapping (routing table). It sends requests directly to the correct node — no intermediate hop.

Lowest latency path. Client must be notified of rebalancing events (e.g. via ZooKeeper watch). Used by: Cassandra driver, Redis Cluster.
Routing Tier

All requests go through a stateless load balancer / router that knows the partition mapping. Clients are simple — they talk only to the router.

Adds one network hop. Routing tier can be updated centrally. Used by: MongoDB mongos, Vitess vtgate.
ZooKeeper Coordination

ZooKeeper (or etcd) acts as the source of truth for partition assignments. Nodes register themselves; routers or clients watch for changes and update their local tables automatically.

Decouples routing metadata from any single node. ZK watches fire immediately on topology change. Used by: HBase, Kafka, SolrCloud.
