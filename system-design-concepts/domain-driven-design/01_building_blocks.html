DDD Building Blocks

Chapter 1 — Entities, Value Objects, Aggregates, Domain Services, and Factories

1.1 — The Problem: Messy Systems Ripple effects and tangled dependencies

A change in one place ripples through 10 others. DDD gives us tools to model domains cleanly — separating concerns so each piece has a single, well-defined responsibility.

1.2 — Entities (Reference Objects) Identity is the defining characteristic

Entities are defined by identity, not attributes. Two entities can share all properties yet remain distinct. Changing an entity's address does not change what entity it is.

1.3 — Value Objects (Immutable Descriptors) Equality by attributes, never mutate — replace

Value Objects have no identity. They are equal if their attributes match. Never mutate — replace. A Money object of $50 USD equals another $50 USD anywhere in the system.

1.4 — Entity vs Value Object Comparison Which one do you need?

If you care about WHICH thing it is, it is an Entity. If you care only about WHAT it is, it is a Value Object. Use identity as your guide.

1.5 — Aggregates Clusters with a root enforcing invariants

An Aggregate is a cluster with a root. All external references point only to the root. Invariants are enforced within — no external code can bypass the root to mutate children.

1.6 — Aggregate Rules & Boundaries Cross-aggregate communication by ID only

Cross-aggregate communication happens by ID, never by object reference. One transaction per aggregate. Eventual consistency ties aggregates together asynchronously.

1.7 — Domain Services Logic that belongs to no single entity

Domain Services contain logic that does not naturally belong to any single Entity or Value Object. The service is stateless; the domain objects carry the data.

1.8 — Factories Encapsulate complex construction

Factories encapsulate the complexity of creating Aggregates and Value Objects, keeping construction logic out of client code and ensuring aggregates are always valid on creation.
