DRY · KISS · YAGNI

Chapter 3 — Everyday Discipline: Principles That Make Code Survivable

3.1 — DRY — Don't Repeat Yourself Every piece of knowledge must have a single, unambiguous representation in the system
DRY: 50%
❌ Duplicated Validation (WET)
// In UserController:
if (email == null || !email.contains("@"))
    throw new BadRequestException("Invalid email");
// In OrderController:
if (email == null || !email.contains("@"))
    throw new BadRequestException("Invalid email");
// In PaymentController:
if (email == null || !email.contains("@"))
    throw new BadRequestException("Invalid email");
✅ Extracted EmailValidator (DRY)
// In EmailValidator:
void validate(String email) {
    if (email == null || !email.contains("@"))
        throw new BadRequestException("Invalid email");
}
// Every controller calls:
emailValidator.validate(email);

// Business rule changes? 1 file. Always.
Files to change (WET) 5
Bugs created 2
Files to change (DRY) 1
Bugs created 0
DRY is about knowledge, not syntax. The goal: a business rule lives in exactly one place. Change it once; the entire system reflects the update.

The Rule of Three: copy once = OK, copy twice = watch it, copy three times = extract it. If you see the same logic in three places, it's time to create a shared function or class.

Be careful though — the wrong abstraction is worse than duplication (Sandi Metz). Don't merge two identical-looking blocks if they represent different business concepts whose future evolution will diverge. Extract only when the knowledge is genuinely the same.

DRY applies far beyond code: database schema, tests, documentation, configuration files, and even deployment scripts. If your staging config and production config duplicate 90% of their values, that's a DRY violation waiting to bite you.

Two blocks of code that look identical but represent different domain concepts should not be merged — their future evolution will diverge and the forced abstraction will make both harder to change. This is why mechanical de-duplication without understanding domain semantics is dangerous.

3.2 — KISS — Keep It Simple, Stupid Prefer the simplest solution that works — not the cleverest
Readability Score: showing complex version
Time to understand 8s
Readability 15%
❌ Clever One-liner
import re
def is_valid_username(u):
    return bool(re.match(
        r'^(?!.*[_.]{2})(?![_.])[a-zA-Z0-9._]{3,20}(?<![_.])$', u
    ))
✅ Readable Multi-step
def is_valid_username(u):
    if len(u) < 3 or len(u) > 20: return False
    if not all(c.isalnum() or c in '._' for c in u): return False
    if u[0] in '._' or u[-1] in '._': return False
    if '..' in u or '__' in u or '._' in u or '_.' in u: return False
    return True
Complexity hides bugs. The "clever" regex is a single point of failure; one wrong character breaks all 5 rules invisibly. The multi-step version lets you test, read, and debug each constraint independently.

Ask yourself: "Would a new team member understand this in 30 seconds?" If no, simplify — not because they're not smart, but because you'll be that new team member 6 months from now when you've forgotten all the context.

Cleverness is usually a sign that the solution needs more thought, not less. The simplest solution often requires the most mental effort to arrive at.

"Any fool can write code that a computer can understand. Good programmers write code that humans can understand." — Martin Fowler

Simple is not the same as easy to write. The simplest solution often requires the most thought. Cleverness = technical debt: someone will eventually need to understand, modify, or debug that regex at 2am during an incident.

KISS applies at architecture level too: 3 services doing clear things beats 10 microservices with ambiguous boundaries. Complexity compounds — every clever shortcut in your codebase multiplies the cognitive load for everyone who follows.

3.3 — YAGNI — You Aren't Gonna Need It Implement things when you actually need them, never when you just foresee you might need them
Codebase weight from speculative features
Plugin system (+200 lines) 0 users
Config flags (+150 lines) always false
Event system (+300 lines) 0 consumers
❌ Speculative Generality
class OrderService {
    // "Might need this for microservices later"
    private final EventBus eventBus;
    private final PluginRegistry plugins;
    private final CacheStrategy cacheStrategy; // not used
    void save(Order o) {
        plugins.runHooks("pre-save", o);
        db.insert(o);
        eventBus.publish(new OrderSaved(o));
        cacheStrategy.invalidate(o.id()); // never called
    }
}
✅ Only What's Needed Today
class OrderService {
    private final OrderRepository db;
    void save(Order o) {
        db.insert(o);
    }
}

// When you ACTUALLY need events → add it then.
// The interface will be cleaner because you'll
// have a real use case to design against.
The speculative code has 3 dependencies that will never be used and 650+ dead lines — all of which need reading, testing, and maintaining forever.

YAGNI saves you from yourself. Requirements change — the feature you built speculatively is rarely the feature that's actually requested. You've now maintained dead code and must build the real thing anyway.

The simplest heuristic: only build something when there's a concrete user story or test that requires it. "We might need it later" is not a requirement.

OCP/YAGNI tension: The Open/Closed Principle says "design for extension." YAGNI says "don't extend prematurely." Resolution: extend when you have the second concrete use case, not in anticipation of it.

The first user defines the interface; the second user validates whether the abstraction is right. Before the second user exists, any abstraction you create is speculation — and speculation is usually wrong because you're missing the information that would make it correct.

Dead code is not neutral — it adds cognitive load to every reader, makes refactoring harder, and can hide bugs. Code that doesn't exist has zero maintenance cost.

3.4 — Boy Scout Rule Always leave the code cleaner than you found it
30 seconds
Renamed d → daysSinceLastLogin while fixing adjacent bug
Permanent clarity for every future reader
5 minutes
Extracted 15-line block into calculateDiscount() while adding nearby feature
Readable forever, testable in isolation
10 seconds
Deleted commented-out oldProcessOrder() while touching the file
Less noise, less confusion about what's active
Each micro-cleanup takes seconds. Compounded across a team over months, the codebase improves organically without needing dedicated "refactoring sprints."

Robert C. Martin (Uncle Bob) coined this from the Boy Scout motto. The idea: you don't need a separate "cleanup week" — just commit to leaving each file slightly better than you found it.

Even tiny improvements compound: a renamed variable today means faster comprehension for everyone who reads that file for the next 5 years. That's a huge ROI on 30 seconds of effort.

Key constraint: "in the area you're already touching." Boy Scout Rule is NOT a license for large refactors inside a feature PR. Scope it.

A 5-line cleanup = good. A 500-line restructure disguised as a cleanup = dangerous and hard to review. Your reviewer cannot distinguish the intentional behavior change from the incidental refactoring — and large structural changes in feature PRs have a way of introducing subtle bugs.

When you identify a large cleanup that needs to happen: open a separate PR, give it a clear name, and make it purely mechanical with no behavior change. Your team will love you for the separation.

3.5 — Fail Fast Surface errors at the earliest possible moment — close to the source
Hops before failure 10
Debug time 2 hrs
❌ Fail Slow (NPE at layer 4)
void processOrder(String userId, int quantity) {
    // Bad input travels all the way to DB
    User user = userService.find(userId);  // null if invalid
    Product p = inventory.get(null);       // NPE here!
    p.reserve(quantity);
}
✅ Fail Fast (validated at border)
void processOrder(String userId, int quantity) {
    Objects.requireNonNull(userId, "userId required");
    if (quantity <= 0)
        throw new IllegalArgumentException("quantity must be positive");
    User user = userService.find(userId)
        .orElseThrow(() -> new NotFoundException(
            "User not found: " + userId));
    inventory.get(user.productId()).reserve(quantity);
}
The golden rule: validate at the border of your system (HTTP endpoints, CLI args, file inputs, external API responses). Assert invariants everywhere else inside. Bad data should never travel far.

Think of validation like a border checkpoint: every piece of external input must be cleared before it enters your system. Once inside, code should be able to assume data is valid — this is what makes logic clean and testable.

When something does go wrong, a Fail Fast error message like "userId required at line 12" pinpoints the problem immediately. A NullPointerException 10 layers deep requires archaeology.

Fail-fast is offensive programming. Know when defensive is needed: user-facing input (never crash the UI), payment flows (validate everything, fail gracefully with a human-readable message), external APIs (always handle failure paths).

Offensive programming is about surfacing bugs loudly in controlled environments — tests, staging, dev — not about crashing production. In production, offensive assertions should produce clear error messages and structured logs that route to your on-call system, not silent NPEs that propagate into data corruption.

assert vs. throw: Use assert for invariants you control (internal consistency checks). Use throw with a meaningful type for conditions driven by external input.
