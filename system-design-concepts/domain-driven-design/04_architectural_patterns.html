Architectural Patterns in DDD

Chapter 4 - Hexagonal architecture, ports/adapters, CQRS, event sourcing, and snapshots.

4.1 -Hexagonal Architecture OverviewDomain at center, adapters around it

Keep the domain independent from frameworks and infrastructure details.

4.2 -Ports (Interfaces)Domain contracts

Ports define required capabilities while hiding implementation technology.

4.3 -Adapters (Implementations)Swap DB adapters safely

Adapters are replaceable; domain behavior remains unchanged.

4.4 -CQRS SplitCommand side and query side

Writes optimize correctness; reads optimize latency and projection shape.

4.5 -CQRS Event FlowProjection updates read model

Read model follows events and serves low-latency queries.

4.6 -Event SourcingStore events, replay state

Event sourcing keeps a full history, not only the latest row values.

4.7 -Event Sourcing vs TraditionalOverwrite row vs append log

Append-only logs improve auditability and temporal debugging.

4.8 -SnapshotsCheckpoint event streams

Snapshots reduce replay time in long-running event streams.
