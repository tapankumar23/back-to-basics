Must-Know Problems

  • 1
    Two Sum LC 1
    HashMap O(n) - store complement to index. Single pass.
  • 2
    3Sum LC 15
    Sort + two-pointer. Fix i, move L and R inward. Skip duplicates at both levels carefully.
  • 3
    Container With Most Water LC 11
    Two-pointer greedy - always move the shorter side inward to maximize potential area.
  • 4
    Merge Intervals LC 56
    Sort by start. Merge if last.end >= curr.start, extend end to max of both.
  • 5
    Insert Interval LC 57
    Find insertion point, merge overlapping neighbors in one pass.
  • 6
    Product of Array Except Self LC 238
    No division. Left-prefix pass then right-suffix pass. O(n) time, O(1) extra space.
  • 7
    Rotate Image LC 48
    Transpose matrix then reverse each row. In-place O(1) space.
  • 8
    Set Matrix Zeroes LC 73
    Use first row/col as markers. Process inner cells first, then headers last to avoid premature zeroing.
  • 9
    Spiral Matrix LC 54
    Shrink four boundaries (top/bottom/left/right) as you traverse each edge layer.
  • 10
    Majority Element LC 169
    Boyer-Moore voting: one candidate + counter. O(1) space, O(n) time.
  • 11
    Gas Station LC 134
    If total gas >= total cost a solution exists. Start where running sum first hits zero.

Staff-Level Angles

    

      
Q: Merge 10B intervals from distributed sensors arriving out of order - won't fit in memory?

      
External sort-merge: spill sorted runs to disk, merge via k-way merge with a min-heap. For streaming, maintain an active-set keyed by sensor ID, merge overlapping windows eagerly, evict closed intervals to cold storage. Partition by time bucket across nodes, reduce per shard, final merge pass. Key insight: you only need to buffer intervals within max_duration of the current watermark.

    

    

      
Q: Product of array except self for 1B fraud transaction amounts under 1 second?

      
Prefix and suffix arrays precomputed offline - embarrassingly parallelisable across cores with SIMD. For incremental updates use a segment tree with product stored at each node (O(log n) update). To avoid overflow on floating-point amounts use log-space addition: log(product) = sum of log(amount_i), then exp at query time. For integer amounts use modular arithmetic if exact values are needed.
