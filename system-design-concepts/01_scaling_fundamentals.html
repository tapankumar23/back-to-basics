Scaling Fundamentals

Chapter 1 — Scale From Zero To Millions Of Users

1.1 — Single Server to Multi-Tier DNS resolution and layer separation
DNS returns an IP address — the client connects directly to that IP. Splitting into Web, App, and Database tiers lets each scale independently. A failure in one tier does not necessarily bring down the others.
1.2 — Vertical vs Horizontal Scaling Scale-up hits a ceiling; scale-out is unlimited
Vertical scaling (adding CPU/RAM to one machine) hits a hard ceiling — both physical and financial. Horizontal scaling (adding more machines) is theoretically unlimited but requires stateless design and a load balancer.
1.3 — Load Balancer Round-robin routing + health-check failover
The load balancer exposes one public IP — clients never see individual server IPs. Health checks poll servers every few seconds; a server that stops responding is removed from rotation automatically.
1.4 — Database Replication Primary writes, replica reads, automatic failover
Writes always go to the Primary; replicas are read-only copies. Replication is asynchronous — replicas may lag milliseconds behind. On primary failure, a replica is promoted (via sentinel or Raft) and becomes the new primary.
1.5 — Cache Layer (Redis) Cache-aside: HIT returns ~1ms, MISS fetches from DB ~100ms
Hit rate: —
Cache-aside (lazy loading): the app checks the cache first. A HIT returns data in ~1ms. A MISS fetches from DB (~100ms), then writes the result into the cache. Cache invalidation — knowing when to evict stale data — is the hard part.
1.6 — Content Delivery Network (CDN) Edge nodes serve static assets near users
Static assets (images, JS, CSS) are pre-cached at CDN edge nodes worldwide — cutting latency from ~200ms to single digits. Dynamic API responses cannot be cached and must still travel to the origin server on every request.
1.7 — Stateless Architecture Session state moved to a shared store
STATEFUL
Stateful servers pin users to a specific instance — routing to the wrong server returns a 403. Moving sessions to a shared Redis instance means any server can handle any request, enabling true horizontal scaling and zero-downtime deploys.
1.8 — Message Queue (Async Processing) Producers and consumers decoupled via a buffer
Queue depth: 0
The queue absorbs bursts — producers enqueue at any rate without blocking. Consumers pull at a steady, sustainable rate. During a spike the queue depth rises; it gradually falls back to zero as consumers catch up.
1.9 — Database Sharding Horizontal partitioning by shard key
Each shard holds a subset of rows — user_id modulo N routes to the correct shard without scanning others. Adding a shard requires resharding: migrating a fraction of keys to the new node, which is why consistent hashing is preferred at scale.
1.10 — Multi-Data-Center Traffic Routing Geo DNS, regional failover, and active-active resilience
A globally distributed system does not stop at one data center. Geo DNS or latency-based routing sends users to the nearest healthy region; if a region goes dark, traffic shifts to the next closest region while replicated data and stateless services preserve continuity.
