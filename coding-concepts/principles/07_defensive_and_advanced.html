  // SQL in presentation!

              + userId;

  if (user.age > 18) {
    applyAdultContent();  // biz logic mixed in

✅ Separated concerns
// Data Layer
User u = userRepo.findById(id);

// Business Logic Layer
boolean adult =
  userService.canAccessAdultContent(u);

// Presentation Layer
userView.render(u, adult);

// Each layer: one question, one reason to change.
// what data? what rules? how to display?
Rule: Each layer answers exactly one question.
— Data layer: What data exists?
— Business layer: What are the rules?
— Presentation layer: How should it look?

If a "UI change" breaks your database code, you've mixed concerns.
SoC is fractal — it applies at every scale:
— Class level: SRP — one class, one reason to change.
— Architecture level: MVC/MVVM — Model, View, Controller each own one domain.
— System level: Microservices — each service owns exactly one bounded context.

A well-applied SoC makes the system horizontally sliceable: you can swap a data store without touching the UI, and vice versa.
7.2 Defensive vs. Offensive Programming Graceful at the border — loud inside
School Use When Verdict
Defensive User-facing input, public APIs, payment flows, external services Appropriate
Defensive Data from untrusted third-party calls Appropriate
Offensive Internal invariants: "this should never be null here" Appropriate
Offensive Module preconditions inside a trusted system boundary Appropriate
Over-defensive Silently swallowing internal unexpected exceptions Harmful
Hidden cost: over-defensive swallows error
❌ catch(Exception e) { log.warn("oops"); return default; }
→ Layer 2 receives garbage data
→ Layer 3 silently propagates
→ Layer 4 produces wrong result
→ Layer 5: NullPointerException — 5 frames deep!
✅ assert order != null : "processOrder: must not be null";
✅ Crash immediately → 1 frame → obvious root cause
Error propagation depth (bad)
0
Error propagation depth (good)
0
❌ Over-defensive: swallows internal error
try {
  processOrder(order);
} catch (Exception e) {
  // silent failure — garbage propagates
  log.warn("Something went wrong");
  return defaultResponse();
}
✅ Offensive: assert invariants inside
void processOrder(Order order) {
  // Preconditions — crash loudly
  assert order != null
    : "processOrder: order must not be null";
  assert order.isValid()
    : "processOrder: order must be valid";
  // proceed knowing invariants hold
}
Simple rule: Defensive at the system border, offensive inside.

— At the edge (user input, external API): validate gracefully, return errors.
— Inside your own code: if a condition "should never happen," assert loudly so you find the bug on day 1, not when a customer reports silent corruption.
Offensive ≠ reckless crashing in production.

Pair assertions with monitoring and alerting so that a crash surfaces as an immediate alert. Tools like Sentry, Datadog, or PagerDuty turn "loud crash" into "loud alert" — far better than silent data corruption discovered months later.

Also see: Bertrand Meyer's Eiffel language design, where the entire contract system is built to make precondition violations immediately visible.
7.3 Dependency Injection & IoC Declare dependencies — don't create them
Preferred

Constructor Injection

Dependencies declared as final fields, set at construction time. Guarantees object is always fully initialized.

Optional

Setter Injection

Dependency set via a setter method. Use when dependency is optional or may change after construction.

Per-call

Method Injection

Dependency passed directly into a method call. Use when the dependency varies per invocation.

Test with SystemClock (hard-wired)
$ run OrderServiceTest
FAIL: time-sensitive — flaky
expected: 2024-01-15
got: 2026-04-16
Test with FakeClock (injected)
$ run OrderServiceTest
PASS: deterministic
time: 2024-01-15 12:00:00
duration: 0.003s
Coupling (hard-wired new)
0%
Coupling (DI constructor)
0%
❌ Hard-wired dependencies
class OrderService {
  // hard-wired — untestable!
  private EmailService email
    = new EmailService();
  private Clock clock
    = new SystemClock();
  // can't swap in test doubles
}
✅ Constructor injection
class OrderService {
  private final EmailService email;
  private final Clock clock;

  OrderService(EmailService email,
               Clock clock) {
    this.email = email; // declared
    this.clock = clock; // injected
  }
}
DI = don't call new inside a class — receive dependencies from outside.

This makes dependencies swappable. In tests you inject a FakeClock; in production you inject the real SystemClock. The class itself never knows the difference — it just uses the interface.
DI ≠ DI framework.

For small systems, a manual "composition root" (one place at startup that wires everything together) is often cleaner than Spring/Guice/Dagger. Framework value appears at scale when the object graph is too large to wire by hand.

IoC (Inversion of Control) is the broader principle: the framework calls your code, rather than your code calling the framework. DI is one way to achieve IoC.
7.4 Design by Contract (DbC) Preconditions · Postconditions · Invariants

Preconditions

Caller's obligations. What must be true before the method runs. If violated → caller's bug.

require amount > 0

Postconditions

Callee's guarantees. What will be true after the method returns. If violated → callee's bug.

ensure balance >= 0

Class Invariants

Always true for any valid instance. Checked before and after every public method.

invariant balance >= 0

Contract violation flow
❌ transfer(-100) — negative amount
→ balance -= -100 … balance grows!
→ accountStatement() sees unexpected value
→ taxCalc() NullPointerException — 4 frames deep
✅ require amount > 0 — fires at frame 1
✅ PreconditionViolationError: amount must be positive
✅ Root cause obvious — fix caller immediately
❌ No contract — silent corruption
function transfer(amount):
  # no checks
  balance = balance - amount
  # negative amount adds to balance!
  # error surfaces 8 layers downstream
✅ Design by Contract
function transfer(amount):
  require amount > 0           # Precondition
  require balance >= amount   # Precondition

  balance = balance - amount

  ensure balance >= 0         # Postcondition
  # invariant: balance >= 0 always
Think like a legal contract:

— Preconditions = what the caller must bring to the table (their obligation).
— Postconditions = what the callee promises to deliver.
— Invariants = ground rules that are always true.

Even without a DbC library, thinking in these terms forces you to design clearer APIs.
Native DbC support by language:

— Kotlin: require(), check(), ensure() (Kotest)
— Python: assert (disable in optimized builds — use hypothesis for property testing)
— Java: Guava Preconditions, Bean Validation, or manual Objects.requireNonNull
— Eiffel: original language with built-in DbC (Bertrand Meyer, 1986)

DbC also synergizes with property-based testing (QuickCheck, Hypothesis): express postconditions as properties, let the tool find counterexamples.
