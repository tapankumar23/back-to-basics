Code Smells

Chapter 5 — Recognising problems before they metastasize
class OrderProcessor {
// 847 lines...
processOrder() {}
sendEmail() {}
generatePDF() {}
updateInventory() {}
calculateTax() {}
formatAddress() {}
// 37 more methods...
}
⚠ 7 code smells detected

Code Doctor

Code smells are surface indicators of deeper design problems. They don't stop compilation — but they slow every engineer who touches the code.

This chapter covers 7 classic smells, their consequences, and prescriptions.

5.1 — God Class / Long Method One class that knows and does everything
→ Extract Class / Method (File 06)
Cyclomatic Complexity
2
⚡Violation: Class accumulates unrelated methods over time
💥Consequence: Merge conflicts every sprint, no one dares refactor
✅Fix: Extract Class per responsibility — OrderProcessor, EmailSender, PDFGenerator
❌ God Class (12 unrelated methods)
class OrderManager {
  placeOrder() {}
  cancelOrder() {}
  sendConfirmationEmail() {}
  sendShippingAlert() {}
  generateInvoicePDF() {}
  exportToExcel() {}
  calculateTax() {}
  applyDiscount() {}
  updateInventory() {}
  syncWarehouse() {}
  formatAddress() {}
  validateCreditCard() {}
  // 847 lines total
}
✅ Three focused classes
class OrderService {
  placeOrder() {}
  cancelOrder() {}
}

class OrderNotifier {
  sendConfirmation() {}
  sendShippingAlert() {}
}

class OrderDocuments {
  generateInvoice() {}
  exportToExcel() {}
}
// Each < 80 lines
If you need to scroll to see the whole class, it's probably too big. A class should fit comfortably on one screen. If you find yourself writing a comment like // === EMAIL SECTION === inside a class, that's a signal it deserves its own file.
A class with 400 lines that has a single, cohesive responsibility (e.g., a rich domain model) is fine. The smell is multiple responsibilities, not line count. Ask: "how many reasons does this class have to change?" If the answer is more than one, split it. Cyclomatic complexity above ~10 per method is a reliable quantitative trigger.
5.2 — Feature Envy A method more interested in another class's data than its own
→ Move Method (File 06)
0
Lines referencing Customer
2
Lines referencing own class
Click "Show Envy"
⚡Violation: Method accesses 6 fields from a foreign class
💥Consequence: Customer changes break OrderService; logic duplicated across callers
✅Fix: Move calculateDiscount() into Customer — it belongs with the data it uses
❌ Feature Envy
class OrderService {
  calculateDiscount(Customer c) {
    // accessing 6 Customer fields
    if (c.getMemberYears() > 5)
      return c.getBaseRate()
           * c.getLoyaltyMultiplier()
           * c.getTierBonus()
           + c.getReferralCredit();
    return c.getBaseRate();
  }
}
✅ Method moved to its home
class Customer {
  calculateDiscount() {
    // uses own fields — natural
    if (this.memberYears > 5)
      return this.baseRate
           * this.loyaltyMultiplier
           * this.tierBonus
           + this.referralCredit;
    return this.baseRate;
  }
}
Coupling to Customer
0%
If a method uses more data from another class than from its own, it probably wants to live in that other class. Count the dot-accesses: customer.getX() vs. this.getX(). Whichever class has more calls "wins" ownership.
Don't apply this rule blindly to data transfer objects or anemic domain models used at API boundaries. Moving logic into a DTO that crosses service boundaries creates coupling at the wrong layer. The exception: if your DTOs are growing methods, consider making them proper domain objects.
5.3 — Primitive Obsession Using primitives where a domain object belongs
→ Introduce Parameter Object (File 06)
createUser( String email , String name , String phone , String country , int age , boolean isAdmin )
⚠ Compiles fine — crashes at runtime!
⚡Violation: 6 primitives — caller can swap email & name silently
💥Consequence: No compile-time safety; invalid values (negative age, bad email) possible anywhere
✅Fix: Introduce UserCreationRequest — one parameter, validated at construction
❌ Primitive Obsession
void createUser(
  String name,
  String email,
  String phone,
  String country,
  int    age,
  boolean isAdmin
) { ... }

// Caller — spot the bug:
createUser(
  "alice@example.com", // ← email
  "Alice",            // ← name
  "555-0100", "US", 28, false
);
✅ Parameter Object + Value Objects
void createUser(
  UserCreationRequest req
) { ... }

// Named construction — self-documenting
createUser(new UserCreationRequest()
  .name("Alice")
  .email("alice@example.com")
  .phone("555-0100")
  .country("US")
  .age(28)
  .isAdmin(false));
Parameter Risk
High
Imagine moving to a new house and putting everything — furniture, food, clothes, documents — into one giant room labeled "stuff". Finding your passport later is a nightmare. Primitive Obsession is that: using String for everything instead of Email, PhoneNumber, Money. Each type is its own room — things go where they belong, and the compiler guards the door.
A value object (e.g., Email, Money) validates at construction time — you can never hold an invalid Email object. This shifts validation from scattered if-checks across the codebase to a single authoritative place. Bonus: renaming/searching for Email in your IDE is trivial; hunting for every String that might be an email is not.
5.4 — Shotgun Surgery One change requires edits in many unrelated places
→ Move Method, Extract Class (File 06)
Files to edit: 7
SRP version: 1
⚡Violation: Discount logic scattered across 7 files
💥Consequence: Forgot to update OrderHistoryService — production bug in reporting
✅Fix: Consolidate discount logic into DiscountPolicy — one place to change
❌ Logic scattered (7 files)
// OrderService.java
if (tier == "GOLD") rate = 0.15;

// CartService.java
if (tier == "GOLD") rate = 0.15;

// InvoiceService.java
if (tier == "GOLD") rate = 0.15;

// ... 4 more files
// Change GOLD → 0.20?
// Must edit all 7 — easy to miss one
✅ Consolidated DiscountPolicy
class DiscountPolicy {
  rateFor(CustomerTier tier) {
    return switch (tier) {
      case GOLD    -> 0.20;
      case SILVER  -> 0.10;
      default      -> 0.00;
    };
  }
}
// Change rate? One file. Done.
Change Blast Radius
7 files
You know a change is small — "just update the GOLD discount rate" — but you still have to open 6 files to make it. That nagging feeling? That's Shotgun Surgery. The code works, but every change is risky because you might forget a file.
Shotgun Surgery and Divergent Change are mirrors of each other:

Shotgun Surgery — one logical change, many classes to edit.
Divergent Change — one class, many unrelated reasons to edit it.

Both violate SRP but from opposite angles. Section 5.5 covers the other side.
5.5 — Divergent Change One class changed for many unrelated reasons
→ SRP — split by axis of change (File 02)
⚡Violation: ReportGenerator touched by Marketing, Finance, AND IT in the same sprint
💥Consequence: Unrelated changes collide, features fight over the same file, merge hell
✅Fix: Split into ReportFormatter, ReportEmailer, ReportExporter — one axis each
❌ ReportGenerator — 3 change axes
class ReportGenerator {
  // Axis 1: Marketing changes format
  formatAsHTML() {}
  applyBranding() {}

  // Axis 2: Finance changes delivery
  sendByEmail() {}
  scheduleMonthly() {}

  // Axis 3: IT changes export
  exportToPDF() {}
  exportToS3() {}
}
✅ One class per axis
class ReportFormatter {
  formatAsHTML() {}
  applyBranding() {}
}

class ReportEmailer {
  sendByEmail() {}
  scheduleMonthly() {}
}

class ReportExporter {
  exportToPDF() {}
  exportToS3() {}
}
Change Axis Count
0
You open the same file for 3 completely unrelated reasons in a single sprint. A designer wants to change report formatting, finance wants to add a new delivery channel, and IT wants a new export format — and they all land in the same class. That's Divergent Change.
Divergent Change is Shotgun Surgery's mirror:

Divergent Change — many changes, one class.
Shotgun Surgery — one change, many classes.

Divergent Change is resolved by SRP: each class should have one reason to change (one actor who drives its changes). When you identify the "axes of change," you've identified the boundaries.
5.6 — Deep Nesting The Pyramid of Doom — cognitive load stacks like indentation
Nesting depth: 1 → Guard Clauses (File 06)
❌ Nested ifs
if (user != null) { if (user.isActive()) { if (user.hasPermission()) { if (!order.isEmpty()) { if (payment.isValid()) { doWork(); // ← buried! } } } } }
→
Guard Clauses
✅ Early returns
if (user == null) return; if (!user.isActive()) return; if (!user.hasPermission()) return; if (order.isEmpty()) return; if (!payment.isValid()) return; doWork(); // ← immediately visible
⚡Violation: 5 levels of nesting before the happy-path logic
💥Consequence: Reviewers miss branches; adding error handling nests it deeper
✅Fix: Guard Clauses — invert conditions and return early; happy path reads top-to-bottom
Cognitive Complexity
1
The happy path should read straight down the left margin. Guards handle the exceptional cases up top, so the reader reaches the "real" logic without wading through nested braces. Nesting depth > 2 is a consistent warning sign.
Early returns need a consistent style. Some teams prefer a "single exit point" convention (one return per method), others embrace guard clauses. Neither is universally right — but the team must agree. Mixed styles in the same codebase cause more confusion than either alone. Cognitive complexity (SonarQube's metric) is a better measure than cyclomatic complexity for nesting smell.
5.7 — Magic Numbers & Strings Unexplained literals that require a decoder ring to understand
→ Replace Magic Literal (File 06)

❌ What does 3 mean?

if (status == 3) {
  shipOrder();
}
if (retries > 7) {
  escalate();
}
if (role == 99) {
  grantAccess();
}
❓ ❓ ❓ …what team wiki did I miss?
→

✅ Self-documenting constants

if (status == OrderStatus.SHIPPED) {
  shipOrder();
}
if (retries > RetryPolicy.MAX_ATTEMPTS) {
  escalate();
}
if (role == Role.SUPER_ADMIN) {
  grantAccess();
}
IDE auto-complete works. Refactoring is safe.
3
→
static final int SHIPPED = 3
→
enum OrderStatus { SHIPPED }
← IDE-safe, exhaustive, refactorable
⚡Violation: Literal 3 appears 14 times across 6 files
💥Consequence: New engineer changes one instance; misses 5 others — subtle data corruption
✅Fix: Named constant or enum — change the value once, everywhere updates automatically
Confusion Meter
0%
array.length - 1, 0 and 1 in obvious arithmetic, Math.PI — these are not magic numbers. A magic number is one whose meaning isn't clear from context. Ask: "if a new engineer reads this, will they know what this number represents without looking it up?" If not, name it.
A static final int SHIPPED = 3 is better than a bare 3, but an enum is even better: the compiler enforces exhaustive switch coverage, IDEs autocomplete the members, and you can't accidentally pass an invalid value. In strongly-typed languages, prefer enums or sealed types for any domain concept with a finite set of states.
