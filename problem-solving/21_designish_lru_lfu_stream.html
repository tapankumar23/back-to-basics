Design-ish DSA

LRU, LFU, sliding window median, reservoir sampling — data structure design at interview depth

Core Concept

Three families of "design-coded" problems tested at staff level. Each requires combining standard data structures to hit O(1) amortized on all operations.

LRU Cache

O(1) get / put

LinkedHashMap shortcut: Java's LinkedHashMap(cap, 0.75f, true) + override removeEldestEntry — 5 lines. But interviewers always want the manual version.

Manual: Doubly-linked list (DLL) + HashMap. Map gives O(1) node access; DLL maintains recency order (head = MRU, tail = LRU).

  • get: move node to head
  • put (exists): update + move to head
  • put (new, full): remove tail + remove from map + add new to head
LFU Cache

O(1) with three maps

Harder than LRU. Evict least-frequently-used; ties broken by LRU order within the same frequency bucket.

Three structures:

  • keyVal map
  • keyFreq map
  • freqKeys → LinkedHashSet (insertion order = LRU)
  • minFreq int

On get: increment freq, migrate key to next bucket, bump minFreq if old bucket empty. On put (full): evict first key in minFreq bucket; insert new at freq=1, set minFreq=1.

Stream Problems

Median + Reservoir

Sliding Window Median (LC 480): Two heaps (max-heap lower half, min-heap upper half). Add: balance heaps. Remove (element leaving window): lazy deletion via delayMap — mark deleted, fix tops during rebalance. Invariant: size diff ≤ 1, max-heap top ≤ min-heap top.

Reservoir Sampling (k of n unknown): Keep first k items. For item i > k: with probability k/i replace a random reservoir element. After n items, each element has k/n probability of being in the reservoir.

// Reservoir invariant
P(item i in reservoir) = k/n
// Two-heap median invariant
maxHeap.top() <= minHeap.top()
|maxHeap| - |minHeap| <= 1

Java Code

// ===== LRU CACHE — Manual DLL + HashMap =====
class LRUCache {
    private final int capacity;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0,0), tail = new Node(0,0);

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail; tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToHead(node);
        return node.val;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.val = value;
            moveToHead(node);
        } else {
            if (map.size() == capacity) {
                Node lru = removeTail();
                map.remove(lru.key);
            }
            Node node = new Node(key, value);
            addToHead(node);
            map.put(key, node);
        }
    }

    private void addToHead(Node n) { n.prev=head; n.next=head.next; head.next.prev=n; head.next=n; }
    private void removeNode(Node n) { n.prev.next=n.next; n.next.prev=n.prev; }
    private void moveToHead(Node n) { removeNode(n); addToHead(n); }
    private Node removeTail() { Node n=tail.prev; removeNode(n); return n; }

    static class Node { int key, val; Node prev, next; Node(int k,int v){key=k;val=v;} }
}

// ===== LFU CACHE =====
class LFUCache {
    private final int capacity;
    private int minFreq;
    private final Map<Integer,Integer> keyVal = new HashMap<>();
    private final Map<Integer,Integer> keyFreq = new HashMap<>();
    private final Map<Integer,LinkedHashSet<Integer>> freqKeys = new HashMap<>();

    public LFUCache(int capacity) { this.capacity = capacity; }

    public int get(int key) {
        if (!keyVal.containsKey(key)) return -1;
        incrementFreq(key);
        return keyVal.get(key);
    }

    public void put(int key, int value) {
        if (capacity <= 0) return;
        if (keyVal.containsKey(key)) { keyVal.put(key, value); incrementFreq(key); return; }
        if (keyVal.size() == capacity) {
            LinkedHashSet<Integer> minSet = freqKeys.get(minFreq);
            int evict = minSet.iterator().next();
            minSet.remove(evict);
            keyVal.remove(evict); keyFreq.remove(evict);
        }
        keyVal.put(key, value); keyFreq.put(key, 1);
        freqKeys.computeIfAbsent(1, k -> new LinkedHashSet<>()).add(key);
        minFreq = 1;
    }

    private void incrementFreq(int key) {
        int f = keyFreq.get(key);
        keyFreq.put(key, f+1);
        freqKeys.get(f).remove(key);
        if (freqKeys.get(f).isEmpty() && f == minFreq) minFreq++;
        freqKeys.computeIfAbsent(f+1, k -> new LinkedHashSet<>()).add(key);
    }
}

// ===== RESERVOIR SAMPLING =====
public int[] reservoirSample(int[] stream, int k) {
    int[] reservoir = Arrays.copyOf(stream, k);
    Random rand = new Random();
    for (int i = k; i < stream.length; i++) {
        int j = rand.nextInt(i + 1);
        if (j < k) reservoir[j] = stream[i];
    }
    return reservoir;
}

Staff-Level Prompts

Distributed Cache

Design a distributed LRU cache serving 1M req/s across 20 nodes. Where does LRU break down at distributed scale?

→ Local LRU per node is cheap but evicts globally-popular items from cold nodes. Global LRU requires cross-node coordination on every access — too expensive at scale. Redis uses probabilistic eviction: sample 5–10 random keys and evict the LRU among them — approximates true LRU with O(1) coordination. Discuss: consistent hashing for key routing, hot-key replication to multiple nodes, and why exact distributed LRU requires too much coordination (O(nodes) messages per access).

Streaming at Scale

You're sampling 1% of events from a 10M events/sec stream for anomaly detection. Reservoir sampling assumes a single stream. How do you extend to parallel streams?

→ Parallel reservoir sampling: each shard independently samples into a local reservoir of size k/shards; merge with a second-level weighted reservoir pass. For frequency estimation use Count-Min Sketch (sub-linear space, tunable error). When approximate top-K is sufficient, prefer the Space Saving algorithm (tracks top-k counters in O(k) space, O(1) updates) over exact counts. Discuss: when to materialize exact counts into a database vs. serve approximate answers from sketch.
