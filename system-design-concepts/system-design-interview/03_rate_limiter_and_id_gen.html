Rate Limiter & ID Generator

Chapters 4 & 7 — Rate limiting algorithms, Snowflake, UUID, and ticket servers

3.1 — Token Bucket Algorithm Burst-friendly rate limiting with steady refill
Refill rate: 3/s
Tokens accumulate up to a max bucket size. Burst traffic is allowed up to the burst size; sustained traffic is limited to the refill rate. When the bucket is empty, further requests receive 429 Too Many Requests.
3.2 — Leaky Bucket Algorithm Fixed output rate regardless of arrival bursts
Leaky bucket processes requests at a fixed rate regardless of arrival rate. Smooths bursts but may drop requests during sustained overload. Unlike token bucket, it cannot absorb short bursts — output is always at the same steady pace.
3.3 — Fixed Window Counter Simple counters reset per window — boundary exploit
Fixed window allows a burst of 2× the limit at window boundaries. Five requests at t=0.9s plus five at t=1.1s means 10 in one real second — double the limit. Sliding window algorithms fix this by tracking individual request timestamps.
3.4 — Sliding Window Log Rolling window eliminates boundary spikes
Window: 1000ms
Sliding window log counts only requests within the exact rolling window, eliminating boundary spikes. The window moves in real time — old requests exit the left edge and the count drops automatically. More memory-efficient with the sliding window counter approximation.
3.5 — Distributed Rate Limiter Multiple gateways synchronized via central Redis
Each gateway syncs to a central Redis store. Local counters allow a fast path; Redis provides global rate enforcement across the cluster. A brief consistency lag means two nodes can simultaneously allow requests that together exceed the limit — race windows require atomic INCR + EXPIRE operations in Redis.
3.6 — Snowflake ID Generator 64-bit monotonic IDs: timestamp + machine + sequence
41-bit timestamp gives 69 years of IDs from a custom epoch. Sequence field allows 4096 IDs per millisecond per machine. IDs are monotonically increasing, which is friendly to B-tree indexes — inserts go to the rightmost leaf and avoid random page splits.
3.7 — UUID Generation Decentralized uniqueness without a central coordinator
UUIDs trade perfect sortability for decentralization. Any node can generate an ID immediately, which is great for availability, but random distribution hurts index locality and human readability compared with time-ordered or centralized ticket schemes.
3.8 — Ticket Server Monotonic IDs from a centralized sequencer
A ticket server gives beautifully ordered IDs and great index locality, but the central sequencer must be highly available. In practice you add replicas, batching, or segment allocation to avoid turning one counter into a global bottleneck.
