Distributed Problems & Consensus

Partial failures, unreliable clocks, process pauses — and the coordination primitives that tame them: linearizability, ordering, and distributed consensus.

Partial Failures

In distributed systems, some nodes may crash or networks may drop packets while other parts keep running normally — unlike a single computer that either works or doesn't.

5-Node Cluster
HPC vs Cloud Philosophy
HPC (Supercomputer)

Stop on any fault. Specialized hardware with low error rates. Checkpoint & restart from a known-good state.

Any node fails → full stop
Cloud / Internet Services

Commodity hardware fails often. Build software that tolerates partial failures and continues serving requests.

Node fails → route around it
Key insight: You cannot know whether a remote node crashed or is just slow — the network gives no reliable answer.

Crash-Stop

A node crashes and never comes back. Simplest model — others detect it via timeout and route around it.

Crash-Recovery

Nodes may crash and restart. Must persist durable state to disk. Most real-world systems use this model.

Byzantine

Nodes can behave arbitrarily — sending lies, colluding, corrupting data. Rare except in adversarial environments (blockchain, aerospace).

Safety

"Nothing bad happens"

Safety can be violated at a specific moment and cannot be undone. Example: two nodes must never both believe they are the leader simultaneously.

Liveness

"Something good eventually happens"

Liveness requires eventual progress. Example: every request that is not dropped must eventually receive a response.

Consistency Spectrum

Weakest — highest availability, lowest latency Strongest — highest cost, lowest availability
Eventual Consistency

Replicas converge eventually. No ordering guarantees. Maximum availability.

Causal Consistency

Causally related ops appear in order. Concurrent ops may differ. Implementable without coordination.

Linearizability

Total order consistent with real time. Requires cross-node coordination. Strongest guarantee.

Version Vectors as Causality Trackers

Each replica maintains a version vector — one counter per replica. Used to detect concurrent writes in leaderless systems (Dynamo, Riak).

Replica A writes "x=1"
[A:1, B:0, C:0]
Replica B writes "x=2" concurrently
[A:0, B:1, C:0]
Neither dominates → CONFLICT
Must reconcile (last-write-wins, merge, or expose to client)

Linearizability vs Serializability & CAP

These terms are often confused. Linearizability is about single-object real-time ordering; serializability is about multi-object transaction isolation.

Linearizability

A recency guarantee on reads and writes of an individual register. Every operation appears to take effect atomically at a single point in real time.

CAP Theorem

During a network partition, choose Consistency (refuse requests) or Availability (serve potentially stale data). You cannot have both.

Note: "CA" without P is impossible — networks always may partition. Real tradeoff is CP vs AP.

Consensus

Consensus lets a set of nodes agree on a single value despite node failures. It is the fundamental primitive behind leader election, atomic commit, and total-order broadcast.

Termination

Every non-faulty node eventually decides a value (liveness).

Agreement

All non-faulty nodes decide the same value (safety).

Validity

The decided value must have been proposed by some node.

Integrity

Each node decides at most once (prevents double decisions).

FLP Impossibility

Fischer, Lynch & Paterson (1985): in an asynchronous system where even one node may crash, no deterministic consensus algorithm can guarantee termination (liveness) in all cases. In practice we use timeouts, randomization, and partial synchrony assumptions to work around this.
