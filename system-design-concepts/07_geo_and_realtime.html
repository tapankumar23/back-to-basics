Geo & Realtime Systems

Geo indexing, proximity search, real-time presence, and map rendering basics

7.1 — Proximity Service: Geohash Recursive subdivision of Earth into string-coded cells
Geohash encodes lat/lon into a string — nearby strings mean nearby places (with exceptions at some boundaries). Precision 4 ≈ 40km, precision 6 ≈ 1.2km, precision 8 ≈ 38m. Adjacent cells share a common prefix — a range query becomes a string prefix search.
7.2 — Proximity Service: Quadtree Recursive space partitioning — split when a cell exceeds capacity
Quadtrees split a space into 4 quadrants when a cell exceeds capacity (~100 businesses). Range queries traverse only the quadrants that intersect the search area, pruning the tree early. Build time is O(n log n); query time is O(log n + k) where k is results.
7.3 — Nearby Friends WebSocket + Redis pub-sub for real-time location sharing
Redis pub-sub enables real-time location sharing without polling. Each user has a channel — your friends subscribe to it. Location updates are lightweight (just lat/lon every 30s). WebSockets keep persistent bidirectional connections between users and chat servers.
7.4 — Google Maps: Routing Dijkstra/A* pathfinding with live traffic rerouting
Google Maps pre-computes shortest paths on road segments and uses bidirectional A* with hierarchical road levels. Real-time traffic from anonymized user GPS data recalculates ETAs continuously. Congestion turns a road segment red — the router finds an alternate path automatically.
7.5 — Google Maps: Map Tiles Zoom levels, tile IDs, CDN caching strategy
Zoom: 0
At zoom 0, 1 tile covers Earth. Each zoom level quadruples tile count (4^z tiles total). CDN caches the billions of pre-rendered static tiles globally. Dynamic data (traffic, satellite) is rendered server-side and overlaid. Tile ID format: {zoom}/{x}/{y}.png.
