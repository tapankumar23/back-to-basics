SOLID Principles

Chapter 2 — Five principles for object-oriented design that produce maintainable, extensible code

SSingle Resp. OOpen/Closed LLiskov Sub. IInterface Seg. DDependency Inv.
2.1 — Single Responsibility Principle A class should have only one reason to change
Change impact:
✗ Before SRPclass UserManager { void authenticate(String user, String pass) { ... } void sendWelcomeEmail(String email) { ... } Report generateMonthlyReport() { ... } String formatDate(Date d) { ... } // 4 stakeholders can ask you to change this! }
✓ After SRPclass AuthService { void authenticate(String u, String p) { ... } } class EmailService { void sendWelcomeEmail(String email) { ... } } class ReportGenerator { Report generateMonthlyReport() { ... } }
ull; New to this? 🔵 Go deeper
One reason to change: If Marketing AND Finance AND IT can all independently ask you to change a class, it has too many responsibilities. Each "stakeholder" = one reason = one class. Ask yourself: who owns the change request for each method?
SRP is fractal: it applies at method level (does one thing), class level (one responsibility), module level (one domain), service level (one bounded context). Zoom in or out and the principle holds. A class with 500 lines that handles only authentication still satisfies SRP.
SRP is about cohesion: grouping things that change together. If a class has multiple responsibilities, changes to one responsibility can break or require re-testing of unrelated functionality. The goal is stable, predictable change boundaries.
2.2 — Open / Closed Principle Open for extension — closed for modification
✗ Violates OCPclass PaymentProcessor { void process(Payment p) { if (p.type.equals("stripe")) chargeStripe(p); else if (p.type.equals("paypal")) chargePayPal(p); // Adding Apple Pay = modifying existing class } }
✓ Follows OCPinterface PaymentProvider { void charge(Payment p); } class StripeProvider implements PaymentProvider { ... } class PayPalProvider implements PaymentProvider { ... } class ApplePayProvider implements PaymentProvider { ... } // New file only - zero changes elsewhere
ull; New to this? 🔵 Go deeper
The USB port analogy: Your laptop has a USB port — it does not need to be redesigned every time a new USB device is invented. The interface is fixed (closed for modification), but any device can plug in (open for extension). Design your classes the same way.
Do not over-engineer early: OCP does not mean "always abstract everything." Premature abstraction before you know the variation axis violates YAGNI. The first payment type does not need an interface. The second one triggers the refactoring. The pattern only applies along the axis where variation actually occurs.
OCP reduces the risk of regressions. When you add behavior by writing new code (not modifying existing code), you cannot break existing tests. The existing code path is unchanged and still verified.
2.3 — Liskov Substitution Principle Subtypes must be substitutable for their base types
Rectangle / Square LSP Violation:
✗ LSP violationclass Penguin extends Bird { void fly() { throw new UnsupportedOperationException(); // Caller expects fly() to work! } } // makeBirdFly(Bird b) crashes with Penguin
✓ Follows LSPclass FlyingBird extends Bird { abstract void fly(); } class NonFlyingBird extends Bird { } class Eagle extends FlyingBird { ... } class Penguin extends NonFlyingBird { ... } // makeBirdFly(FlyingBird b) is type-safe, no crash
ull; New to this? 🔵 Go deeper
Quick litmus test: If caller code needs instanceof to check what subtype it received before calling a method, you are violating LSP. The whole point of polymorphism is that callers do not need to know which concrete type they are working with.
Behavioural subtyping (Barbara Liskov, 1987): Three formal rules — (1) Preconditions cannot be strengthened by subclass. (2) Postconditions cannot be weakened. (3) Invariants of the superclass must hold. These are the contract clauses a subclass must honour.
LSP violations surface as runtime exceptions, unexpected null returns, or silent wrong results. They compile fine — the type system cannot catch them. Only tests and runtime execution reveal the violation.
2.4 — Interface Segregation Principle Clients should not depend on methods they do not use
✗ Fat interfaceinterface IWorker { void work(); void eat(); void sleep(); void attendMeeting(); } class Robot implements IWorker { public void eat() { throw new UnsupportedOperationException(); } }
✓ Segregatedinterface IWorkable { void work(); } interface IFeedable { void eat(); } interface ISleepable { void sleep(); } class Robot implements IWorkable { public void work() { ... } // no forced eat() or sleep() } class Human implements IWorkable, IFeedable, ISleepable { ... }
ull; New to this? 🔵 Go deeper
Restaurant menu analogy: A tasting menu with 5 items is clear and focused. A single menu with breakfast, lunch, dinner, catering, drinks, and events is overwhelming and forces you to flip past irrelevant sections. Fat interfaces force implementors to deal with irrelevant methods.
ISP and SRP are deeply related: If your interface is fat, the implementation class probably violates SRP too. Fix SRP first — ISP often fixes itself. A fat interface is usually a symptom, not the root cause. After segregation, changing IFeedable does not require Robot to be recompiled.
ISP reduces coupling. When a class depends on a narrow interface, changes to unrelated methods cannot affect it. This is especially important at module boundaries where recompilation cascades become costly.
2.5 — Dependency Inversion Principle Depend on abstractions, not concretions
✗ Violates DIPclass OrderService { private MySQLRepository db = new MySQLRepository(); // hard dependency void save(Order o) { db.insert(o); } } // Cannot swap DB without editing OrderService
✓ Follows DIPclass OrderService { private final IOrderRepository db; OrderService(IOrderRepository db) { this.db = db; // injected } void save(Order o) { db.save(o); } } // Swap any IOrderRepository impl, zero edits here
ull; New to this? 🔵 Go deeper
Power outlet analogy: Your laptop does not depend on a specific power plant. The wall outlet is the abstraction both sides depend on. DIP works the same way — define an interface (the outlet standard), then both high-level and low-level components depend on it.
DIP is not Dependency Injection: DIP is the design principle — depend on abstractions. Dependency Injection (DI) is one mechanism to achieve it. You can follow DIP with manual wiring in a main() method — no framework required.
DIP inverts the traditional layering direction. Instead of business logic importing infrastructure classes, both sides import a shared abstraction. This makes business logic portable and independently testable — inject an InMemoryRepository in tests with no database needed.
