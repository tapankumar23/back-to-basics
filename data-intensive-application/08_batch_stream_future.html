Hadoop / MapReduce

Same pipeline logic, but each stage is a MapReduce job. Intermediate output is materialized to HDFS before the next stage — enabling fault tolerance by re-running failed stages from the last checkpoint.

🔒 Immutable Input

The original input dataset is never modified. You can re-run any stage or debug by replaying from any intermediate file. This is the foundation of reproducible batch processing.

MapReduce Internals

Map phase emits (key, value) pairs. Shuffle routes pairs to the correct reducer by key hash. Reducers sort and aggregate each key group.

MapReduce Join Strategies

Three strategies trade off between shuffle cost, memory usage, and pre-processing requirements.

Reduce-Side Join

Both datasets A and B are shuffled by join key to the same reducer. Works for any dataset sizes but requires a full shuffle — the most expensive network operation in MapReduce.

Broadcast Hash Join

Small dataset B is loaded into each mapper's memory. Each mapper joins against the in-memory hash. No shuffle needed — fastest when one dataset fits in RAM.

Partitioned Map-Side Join

Both datasets are pre-partitioned and pre-sorted by join key. Each mapper joins its local co-located partitions — no shuffle, no extra memory. Fastest but requires pre-processing.

Graph & Iterative Processing (Pregel)

Pregel's Bulk Synchronous Parallel model: nodes exchange messages each superstep, separated by a global barrier. Vertex state stays in memory — far faster than MapReduce disk I/O.

PageRank per node

💾 Why MapReduce Struggles with Graphs

Each MapReduce iteration must write all vertex state to disk and re-read it — even if only a few nodes changed. For 50 iterations on a billion-node graph, that is 50x disk I/O. Pregel keeps vertex state in memory across supersteps.

Dataflow Engines (Spark / Flink)

Operator DAGs replace the two-stage map/reduce split. Data flows in-memory between operators — no intermediate materialization.

MapReduce writes intermediate files to disk between each stage. A 5-stage pipeline requires 4 full disk round-trips before the final result.

Fault Tolerance via Lineage

Spark tracks the lineage graph of transformations. On node failure, only the lost partition is recomputed from its upstream source — no need to checkpoint all intermediate state to disk.

Log Compaction

For changelog-style topics, Kafka retains only the latest value per key. Old entries are replaced during compaction. A compacted log equals the full current state — not the full history.

Keeping Systems in Sync

Dual-write: writing to two systems independently creates a race condition. If one write fails, the systems permanently diverge.

DB 📂
Search 🔍

CDC as the fix: Write only to the DB. A CDC consumer reads the DB replication log and propagates changes to all downstream systems in order — a single ordered source of truth with guaranteed delivery.

Change Data Capture (CDC)

CDC taps into the database write-ahead log to stream every INSERT, UPDATE, and DELETE to downstream consumers — without touching application code.

Schema Evolution

When you ALTER TABLE ADD COLUMN, Debezium-style CDC captures before/after images. Downstream consumers receive both old and new schema events and can handle them independently.

CDC vs Dual-Write

CDC: DB log is the single writer. All downstream updates are after-the-fact, ordered, and replayable.

Dual-write: App writes to two systems independently. Partial failure = permanent divergence.

Exactly-Once Semantics

Delivery guarantees in distributed systems — and how to achieve exactly-once despite failures and retries.

At-Most-Once

Fire and forget. If the send fails, the message is lost — no retry.

At-Least-Once

Retry on failure. Message delivered, but possibly duplicated.

Exactly-Once

Idempotent dedup: duplicate detected by dedup key, dropped at destination.

Distributed Transactions in Streams

Kafka Streams and Flink achieve exactly-once by committing the output offset and the external DB write atomically. If either fails, both are rolled back and the input is reprocessed from the last committed offset.

The Future: Unbundling Databases

Decompose the monolithic database into specialized components wired together by event streams.

Dataflow Architecture

No central DB. Each service owns its derived view, updated by subscribing to event streams. The stream is the source of truth.

These two properties can be decoupled. Eventual consistency for timeliness (stale reads are OK) while enforcing strong integrity (no data loss, no corruption).
