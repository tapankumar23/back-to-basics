Refactoring Patterns

Targeted transformations that improve code structure without changing observable behavior — each one a precise tool with a name.

6.1 Extract Method — when a comment explains a block, make that comment the method name
⚠️ Violation: 60-line method mixing validation, calculation, email, and inventory in one body. A comment before each block screams "I should be a method."
💥 Consequence: Hard to test in isolation, impossible to reuse a sub-task, code review is exhausting, merge conflicts are frequent.
✅ Fix: Name the intent, extract the block. processOrder becomes a readable table of contents — five lines that each delegate.
❌ Before
void processOrder(Order order) {
  // validate items
  if (order.items.isEmpty())
    throw new IllegalStateException();
  checkStockLevels(order.items);
  // calculate totals
  order.subtotal = order.items.stream()
    .mapToDouble(Item::getPrice).sum();
  order.tax = order.subtotal * 0.08;
  // apply discounts
  if (order.promoCode != null)
    applyPromoCode(order);
  // send confirmation (12 lines)
  // update inventory (15 lines)
  // ... 60 lines total
}
✅ After
void processOrder(Order order) {
  validateItems(order);
  calculateTotals(order);
  applyDiscounts(order);
  sendConfirmationEmail(order);
  updateInventory(order);
}

// Each sub-method is 5-15 lines,
// named, testable, reusable.
Method Complexity
High
Rule: If you need a comment to explain what a block of code does, extract it into a method and make that comment the method name. The code now documents itself.
Nuance: The real cost of extraction is naming. If you cannot give the extracted block a clear, intent-revealing name, the block itself may not be cohesive — it may need to be split further, or it truly belongs inline. A bad extracted method name (doStuff2) is worse than the original comment.
6.2 Guard Clauses — fail fast at the top so the happy path reads straight down
if (user != null) {
if (user.isActive()) {
if (user.hasPermission()) {
doWork();
}
}
}
if (user == null) return;
if (!user.isActive()) return;
if (!user.hasPermission()) return;
doWork();
Nesting depth: 3
⚠️ Violation: Three nested if blocks create an "arrow" or "pyramid" shape — the happy path hides at the deepest indentation level.
💥 Consequence: Readers must mentally track all enclosing conditions simultaneously. Each additional nesting level doubles the cognitive load.
✅ Fix: Invert each condition to an early return. Edge cases are handled at the top; the happy path flows straight down the left margin.
❌ Arrow Pattern
if (user != null) {
  if (user.isActive()) {
    if (user.hasPermission()) {
      doWork();
    }
  }
}
✅ Guard Clauses
if (user == null)           return;
if (!user.isActive())      return;
if (!user.hasPermission()) return;

doWork();
Nesting Depth
3
Rule: The happy path should read straight down the left margin. Handle edge cases at the top with early returns so the main logic stands alone without wrapping conditions.
Nuance: Some teams enforce a "single exit point" convention — every function has exactly one return. Guard clauses directly conflict with that rule. Neither is universally right. Guard clauses tend to win in modern Java/Kotlin/TypeScript style guides because readability outweighs theoretical purity of single-exit.
6.3 Replace Conditional with Polymorphism — adding a case should mean adding a file, not modifying one
Click "Add Triangle" to see how each design handles extension
⚠️ Violation: A switch on a type tag. Every new shape requires opening the existing method and modifying it — violating Open/Closed Principle.
💥 Consequence: The switch grows forever. Miss one case in one method, and the bug appears at runtime. The same type check is scattered across the codebase.
✅ Fix: Define an interface with the behavior. Each shape implements it. Adding Triangle = creating a new class — existing code is never modified.
❌ Type Switch
double area(Shape shape) {
  switch (shape.type) {
    case "circle":
      return Math.PI * r * r;
    case "square":
      return side * side;
    case "triangle": // must edit here
      return 0.5 * base * height;
  }
}
✅ Polymorphism
interface Shape {
  double area();
}
class Circle implements Shape {
  double area() { return PI*r*r; }
}
class Square implements Shape {
  double area() { return side*side; }
}
// Triangle = new file, zero edits
OCP Violation Score
High
Rule: When you need to add a new case, you should add a new file, not modify an existing one. This is the Open/Closed Principle applied directly.
Nuance: Small, stable switch statements — 2–3 cases that will never grow — are perfectly fine. Polymorphism only pays off when the axis of variation is expected to grow. Premature polymorphism adds classes for problems that do not exist yet. Ask: "Will I realistically add more cases?" before extracting.
6.4 Introduce Parameter Object — when parameters travel together, they belong in a class
keyword: String
from: Date
to: Date
page: int
pageSize: int
sortBy: String
SearchCriteria
.keyword
.dateRange   // DateRange(from, to)
.pagination   // Pagination(page, size)
.sortBy
⚠️ Violation: Six primitive parameters passed to search(). The same six appear on every related method: searchByTag(), searchByAuthor(), exportResults()...
💥 Consequence: Change the sort order field name — update 12 call sites. Validate date range logic — duplicate it in every method. Parameters arrive in the wrong order at call sites.
✅ Fix: Group the parameters into SearchCriteria. All methods share one type. Validation lives inside the class. Adding a field changes one place.
❌ Primitive Obsession
List<Result> search(
  String keyword,
  Date from,
  Date to,
  int page,
  int pageSize,
  String sortBy
) { ... }
✅ Parameter Object
List<Result> search(
  SearchCriteria criteria
) { ... }

// SearchCriteria validates itself:
// dateRange.from <= dateRange.to
Parameter Count
6 params
Rule: When you see the same group of parameters appearing together in multiple methods, that group wants to become a class. Give it a name — the name will reveal the concept you were missing.
Nuance: Parameter objects often evolve into proper value objects — immutable, self-validating types like DateRange (ensures from ≤ to) or Money (enforces currency consistency). The next step after introduction is encapsulation: move behavior that operates on the data into the object itself.
6.5 Move Method — methods should live where the data they need lives
⚠️ Violation: OrderService.calculateDiscount() reaches into 5 fields on Customer — loyalty tier, account age, promo eligibility, region, subscription status.
💥 Consequence: Feature Envy. Every change to Customer's internal structure forces changes in OrderService. Objects are coupled across class boundaries.
✅ Fix: Move calculateDiscount() to Customer. All the data it needs is already there. Coupling becomes internal; callers use a clean API.
❌ Feature Envy
class OrderService {
  double calculateDiscount(Customer c) {
    // uses c.loyaltyTier
    // uses c.accountAge
    // uses c.promoEligible
    // uses c.region
    // uses c.subscription
    ...
  }
}
✅ Method Lives with Data
class Customer {
  double calculateDiscount() {
    // this.loyaltyTier
    // this.accountAge
    // this.promoEligible
    // all local, no coupling
  }
}
// caller: customer.calculateDiscount()
Feature Envy Score
High
Rule: Ask "what data does this method use most?" The answer tells you where the method belongs. If it uses more data from another class than its own, move it there.
Nuance: Moving a method requires updating all callers — a rename + delegation chain. This is exactly where automated refactoring tools (IntelliJ "Move Method", VS Code rename symbol) shine: they do the update atomically across the whole codebase, making it safe to refactor without manual find-and-replace errors.
6.6 Rename for Intent — the cheapest, highest-leverage refactoring
int d;   boolean f;
// reference 1: d > 30 ? sendReminder() : null;
// reference 2: f == true ? sendVerification() : skip();
void proc (List<int[]> l) { ... }
// reference 3: calling proc (rawData);
⚠️ Violation: Single-letter variables (d, f), cryptic method names (proc), meaningless parameter names (l). Every reader must decode intent from scratch.
💥 Consequence: High cognitive load per line. Context-switching cost is enormous. Code reviews slow down. Bugs hide in unfamiliar symbols. New team members take weeks to orient.
✅ Fix: Rename — it is free. d → daysSinceLastLogin, f → isEmailVerified, proc → getFlaggedCells. Names are documentation that never goes stale.
❌ Cryptic Names
int d;
boolean f;
void proc(List<int[]> l) {
  for (int[] a : l) {
    if (a[0] == 4)
      flagged.add(a);
  }
}
✅ Intent-Revealing Names
int daysSinceLastLogin;
boolean isEmailVerified;
void getFlaggedCells(
    List<Cell> cells) {
  for (Cell cell : cells) {
    if (cell.status == STATUS_FLAGGED)
      flagged.add(cell);
  }
}
Cognitive Load
Very High
Rule: If you have to think for even a second about what a variable or method name means, rename it. Clarity is not a luxury — it is the primary engineering output.
Key insight: Renaming is the cheapest, highest-leverage refactoring available. It requires zero architectural changes, zero risk of behavior change, and tools do it atomically. Yet it is often skipped because it feels cosmetic. It is not cosmetic — good names are documentation that lives with the code and is updated automatically every time the code moves. A comment describing "d" can go stale the moment someone changes what "d" stores. The name "daysSinceLastLogin" cannot.
