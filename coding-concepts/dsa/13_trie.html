PATTERN 13

Trie (Prefix Tree)

Character-by-character tree enabling O(L) prefix queries impossible with hashing. Trade memory for prefix power.

When to Use

Autocomplete / Prefix Search
Find all words sharing a prefix in O(prefix_len + results). HashMaps can't do this without scanning all keys.
Word Existence Checks
insert/search/startsWith all O(L). Wildcards ('.'​) handled via DFS on trie nodes.
XOR Trie
Insert integers bit-by-bit MSB→LSB. Query opposite bit greedily to maximize XOR in O(32).
Memory note: Each node holds 26 child pointers (or a HashMap). A trie for 10M English words can consume several GB. Consider compressed trie / radix tree or DAWG when memory matters.

Java Implementation

// TrieNode — 26-child array (faster) or HashMap (sparse/unicode)
class TrieNode {
    TrieNode[] c = new TrieNode[26];
    boolean end = false;
}

class Trie {
    TrieNode root = new TrieNode();

    // O(L) insert
    public void insert(String w) {
        TrieNode cur = root;
        for (char ch : w.toCharArray()) {
            int i = ch - 'a';
            if (cur.c[i] == null) cur.c[i] = new TrieNode();
            cur = cur.c[i];
        }
        cur.end = true;
    }

    // O(L) exact search
    public boolean search(String w) {
        TrieNode n = node(w);
        return n != null && n.end;
    }

    // O(L) prefix check
    public boolean startsWith(String p) { return node(p) != null; }

    private TrieNode node(String s) {
        TrieNode cur = root;
        for (char ch : s.toCharArray()) {
            int i = ch - 'a';
            if (cur.c[i] == null) return null;
            cur = cur.c[i];
        }
        return cur;
    }
}

// Word Search II — trie-based board DFS
public List<String> findWords(char[][] board, String[] words) {
    Trie trie = new Trie();
    for (String w : words) trie.insert(w);
    Set<String> res = new HashSet<>();
    int R = board.length, C = board[0].length;
    for (int r = 0; r < R; r++)
        for (int c = 0; c < C; c++)
            dfs(board, r, c, trie.root, "", res);
    return new ArrayList<>(res);
}
void dfs(char[][] b, int r, int c, TrieNode node, String path, Set<String> res) {
    if (r < 0 || r >= b.length || c < 0 || c >= b[0].length || b[r][c] == '#') return;
    int i = b[r][c] - 'a';
    if (node.c[i] == null) return; // prune — no trie path
    node = node.c[i]; path += b[r][c];
    if (node.end) res.add(path);
    char tmp = b[r][c]; b[r][c] = '#';
    dfs(b,r+1,c,node,path,res); dfs(b,r-1,c,node,path,res);
    dfs(b,r,c+1,node,path,res); dfs(b,r,c-1,node,path,res);
    b[r][c] = tmp;
}

// Maximum XOR of Two Numbers (XOR Trie)
int[][] xorTrie = new int[32*100001][2];
int xorIdx = 1;
void xorInsert(int n) {
    int cur = 0;
    for (int b = 31; b >= 0; b--) {
        int bit = (n >> b) & 1;
        if (xorTrie[cur][bit] == 0) xorTrie[cur][bit] = xorIdx++;
        cur = xorTrie[cur][bit];
    }
}
int xorQuery(int n) {
    int cur = 0, result = 0;
    for (int b = 31; b >= 0; b--) {
        int want = 1 - ((n >> b) & 1); // prefer opposite bit
        if (xorTrie[cur][want] != 0) { result |= (1 << b); cur = xorTrie[cur][want]; }
        else cur = xorTrie[cur][1-want];
    }
    return result;
}

Staff-Level Questions

Q1 — Memory at Scale

Autocomplete trie for 10M English words consumes 8GB RAM. How do you reduce it?

Compressed trie / Radix tree — merge chains of single-child nodes into edge labels, reducing node count dramatically.
DAWG (Directed Acyclic Word Graph) — share suffixes too, not just prefixes. Recognizes equivalent subtrees and merges them.
Ternary Search Tree — 3 pointers per node (less, equal, greater) instead of 26. Cache-friendlier traversal.
Serialized + memory-mapped — serialize trie to compact byte array (trie.bin), mmap it; OS handles paging, no JVM heap pressure.

Q2 — High-Throughput Autocomplete

Autocomplete serves 100k QPS sub-5ms P99 with a 4GB trie. How do you scale to 10M QPS?

Shard by first 2 chars — 676 possible shards, route "app..." to shard for "ap". Each shard holds ~6MB.
Read replicas — trie is read-heavy; replicate across instances. Updates batched hourly via background job.
Pre-materialized top-K per prefix — offline compute top-5 suggestions for every prefix up to length 8. Serve from Redis hash O(1).
Bloom filter — reject prefixes with zero matches before hitting the trie. Eliminates 60%+ of cold-miss lookups.
